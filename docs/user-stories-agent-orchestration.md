# Agent Firewall 智能体编排平台用户故事

## 产品定义

Agent Firewall 面向需要使用智能体完成工作的普通用户和业务人员。用户可以一键导入别人已经写好的 Agent、Skill 和 MCP Tool，把它们拖入流程，配置输入输出、分支、验收和纠正规则，再通过手动、定时或外部事件运行完整自动化流程。

核心价值不是帮助用户开发 Agent，而是让用户安全、直观地复用已有能力，并把多个能力组合成可以重复运行、自动纠错、必要时由人接管的工作流。

## 范围与假设

- 核心用户：流程创建者，不要求会编程或理解内部 JSON。
- 辅助用户：负责凭据、审批、发布和运行维护的平台操作员。
- 首批导入来源：本地文件夹或 ZIP、Git 仓库、Agent Firewall 能力包、MCP Server、受支持协议的远程 Agent。
- Agent、Skill 和 Tool 导入后统一展示名称、用途、输入、输出、版本、来源和健康状态。
- 自动纠正必须有最大次数，涉及写入或外部副作用时必须支持审批或幂等保护。
- 当前设计参考：[桌面界面](../desktop/renderer/index.html) 和 [流程实现](../src/agent_firewall/flow.py)。专用交互稿尚未提供。

每条故事使用 3 C's 结构，并控制为一个 Sprint 内可以独立开发和验收的范围。

## US-01 导入本地 Agent 或 Skill

**Card：** 从文件夹或 ZIP 一键导入能力

**Description：** 作为流程创建者，我希望选择一个本地文件夹或 ZIP 就能导入别人写好的 Agent 或 Skill，以便直接在流程中使用。

**Conversation：** 系统负责识别能力类型、读取说明和检查文件，不要求用户手工修改配置。导入内容应复制到受管理目录，避免原文件移动后流程失效。

**Design：** [能力库入口](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 用户可以选择文件夹或 ZIP，并在一次操作中导入其中检测到的 Agent 和 Skill。
2. 导入前显示能力名称、类型、版本、说明、可执行动作和来源路径。
3. 文件缺失、格式错误或引用越出能力目录时，系统拒绝导入并指出具体问题。
4. 导入成功后，能力出现在能力库中，并可立即加入新流程。
5. 同名同版本能力已存在时，用户可以选择跳过、覆盖或另存为副本。

## US-02 从 Git 仓库导入并更新能力

**Card：** 从 Git URL 导入 Agent 或 Skill

**Description：** 作为流程创建者，我希望粘贴 Git 仓库地址就能导入能力，以便复用团队或社区维护的 Agent 和 Skill。

**Conversation：** 用户可以选择分支、Tag 或 Commit。更新已有能力时，需要先看到版本变化，不能静默改变正在运行的流程。

**Design：** [能力库入口](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 用户输入 Git URL 后，可以选择分支、Tag 或 Commit 并扫描可导入能力。
2. 私有仓库凭据通过受保护的凭据引用提供，不显示在 URL、日志或导出文件中。
3. 导入结果记录仓库地址和 Commit，并在能力库中展示来源版本。
4. 发现新版本时，系统展示变化摘要，并由用户确认是否升级。
5. 升级能力不会修改已发布流程固定使用的旧版本。

## US-03 连接远程 Agent 和 MCP Server

**Card：** 导入远程能力

**Description：** 作为流程创建者，我希望通过连接信息导入远程 Agent 或 MCP Server，以便把外部服务提供的能力加入流程。

**Conversation：** 系统先测试连接并发现可调用契约。MCP Server 只是能力来源，用户实际编排的是发现出的 Tool。

**Design：** [能力配置与 MCP 发现](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 用户可以添加受支持协议的远程 Agent 连接，以及 stdio 或网络 MCP Server 连接。
2. 保存前可以测试连接，并看到成功、认证失败、网络失败或协议不兼容结果。
3. 连接成功后，系统发现并展示可调用 Agent 或 MCP Tool 的名称、说明和输入输出契约。
4. MCP Server 本身不能被拖入流程，只有发现成功的 MCP Tool 可以作为节点。
5. 连接暂时离线时，已保存配置不会被删除，相关能力会被标记为不可用。

## US-04 查看统一能力库

**Card：** 统一管理所有导入能力

**Description：** 作为流程创建者，我希望在一个地方查看所有 Agent、Skill 和 Tool，以便快速找到适合当前任务的能力。

**Conversation：** 不同来源的能力使用同一种展示方式，但保留来源、版本和运行方式。用户应能看懂能力需要什么输入、会产生什么输出。

**Design：** [能力库](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 能力库按 Agent、Skill Action 和 Tool 分类展示可执行能力。
2. 每个能力显示名称、用途、来源、版本、输入、输出和当前健康状态。
3. 用户可以按名称、类型、来源和健康状态搜索或筛选。
4. 不兼容、缺少凭据或发现结果过期的能力不能加入流程，并显示原因。
5. 用户可以从能力详情直接新建流程或添加到当前流程。

## US-05 安全配置连接凭据

**Card：** 一次配置，多处复用凭据

**Description：** 作为平台操作员，我希望安全保存模型、Git 和远程服务凭据，以便流程使用这些能力时不用反复输入密钥。

**Conversation：** 流程只保存凭据引用，不保存明文。普通用户可以知道缺少哪个连接，但不能查看密钥内容。

**Design：** [设置与策略](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 用户可以创建、测试、更新和删除模型、Git、Agent 与 MCP 连接凭据。
2. 凭据保存后只显示名称和掩码，界面不能再次读取完整明文。
3. 流程节点引用凭据名称，流程保存、复制和导出时不包含明文密钥。
4. 运行日志、错误、节点输入输出和事件在持久化前脱敏。
5. 删除仍被流程引用的凭据前，系统列出受影响流程并要求确认。

## US-06 拖拽创建智能体流程

**Card：** 可视化流程编排

**Description：** 作为流程创建者，我希望把 Agent、Skill 和 Tool 拖到画布并连接起来，以便直观看到任务如何一步步完成。

**Conversation：** 画布是产品主工作区。节点代表实际可执行能力，连线代表数据和控制流，用户不需要编辑流程 JSON。

**Design：** [高级编排画布](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 用户可以从能力库拖入 Agent、Skill Action 和 MCP Tool 节点。
2. 用户可以连接、移动、复制、重命名和删除节点，画布不会因节点内容变化而跳动。
3. 流程始终有唯一开始节点和结束节点，系统阻止无效连接和自连接。
4. 画布修改自动保存，重新打开工作区后节点位置和连线保持不变。
5. 用户可以撤销和重做最近的画布操作。
6. 节点和主要画布操作可以仅用键盘完成。

## US-07 配置节点输入和结果传递

**Card：** 用表单连接上下游数据

**Description：** 作为流程创建者，我希望选择上一步的结果作为下一步输入，以便多个能力可以连续协作。

**Conversation：** 输入字段来自能力契约。用户可以填写固定值、引用流程初始输入或引用前序节点输出，原始 JSON 只作为高级选项。

**Design：** [节点配置面板](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 节点配置面板根据能力契约生成文本、数字、开关、选项和对象等输入控件。
2. 每个输入可以使用固定值、流程输入或可到达的前序节点输出。
3. 用户选择输出字段时，可以看到字段名称、类型和示例值。
4. 输入缺失或类型不匹配时，系统在运行前指出节点和字段。
5. 高级 JSON 编辑与表单数据保持同步，切换时合法内容不会丢失。

## US-08 配置分支、并行和汇合

**Card：** 控制流程走向

**Description：** 作为流程创建者，我希望根据节点结果选择不同路线，并让独立任务并行执行，以便表达真实业务流程。

**Conversation：** 常见控制方式包括成功、失败、等待输入、被策略阻止和字段条件。并行分支汇合时，应等待需要的上游节点完成。

**Design：** [流程连线设置](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 用户可以为连线选择成功、失败、等待输入、被阻止或始终执行条件。
2. 用户可以根据节点输出字段添加等于、包含、存在和数值比较条件。
3. 无依赖的分支可以并行运行，汇合节点等待所有被激活的前序分支完成。
4. 未命中的分支被标记为跳过，不会阻止流程正常结束。
5. 无法到达结束节点、存在死路或存在无边界循环时，系统禁止运行。

## US-09 运行前检查整个流程

**Card：** 一键预检流程

**Description：** 作为流程创建者，我希望运行前看到所有配置问题，以便不用运行到一半才发现能力或输入不可用。

**Conversation：** 预检检查流程结构、能力版本、连接、凭据、输入映射、审批和执行限制。问题应能直接定位到画布节点。

**Design：** [流程运行工具栏](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 预检检查流程连通性、节点能力、版本、凭据、输入类型和必要参数。
2. MCP schema 或远程 Agent 契约过期时，预检要求重新发现后再运行。
3. 每个问题显示严重级别、节点、字段、原因和处理建议。
4. 用户点击问题后，画布定位并选中对应节点或连线。
5. 阻断问题未解决时不能运行；警告可以由用户确认后继续。

## US-10 运行并监控自动化流程

**Card：** 启动流程并查看实时进度

**Description：** 作为流程创建者，我希望输入目标后启动流程，并实时看到每个节点状态，以便知道自动化是否正常工作。

**Conversation：** 运行视图需要展示当前节点、输入来源、结果摘要、耗时和下一步。关闭界面后，运行证据仍应保留。

**Design：** [运行面板](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 用户可以填写本次目标和流程输入，然后启动已通过预检的流程。
2. 画布实时显示等待、运行、成功、失败、暂停和跳过状态。
3. 用户可以查看每个节点的输入来源、输出摘要、尝试次数和耗时。
4. 用户可以取消运行，系统停止后续节点并保存明确的取消状态。
5. 用户关闭并重新打开应用后，仍可查看运行状态、事件和最终结果。

## US-11 定义结果验收规则

**Card：** 判断节点或流程结果是否合格

**Description：** 作为流程创建者，我希望为关键结果设置验收规则，以便系统自动判断是否继续、纠正或转人工处理。

**Conversation：** 确定性规则优先，包括状态、字段、文本和 schema。需要主观判断时，可以选择一个评审 Agent，但必须保存判断理由。

**Design：** [节点配置与结果面板](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 用户可以在节点或流程结束处添加状态、字段值、文本包含和 schema 验收规则。
2. 多条规则可以配置为全部通过或任一通过。
3. 用户可以选择评审 Agent，并提供评分标准和最低通过分数。
4. 每次验收保存通过状态、失败规则、实际值和评审理由。
5. 验收失败可以连接到失败路线、自动纠正或人工接管。

## US-12 自动纠正不合格结果

**Card：** 有边界的自动纠正闭环

**Description：** 作为流程创建者，我希望结果不合格时由指定 Agent 自动分析问题并重新处理，以便减少人工反复修改。

**Conversation：** 纠正配置包括纠正 Agent、可用上下文、最大次数和失败去向。每次尝试必须保留原结果、问题和修正内容。

**Design：** [纠正与恢复区域](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 用户可以为验收失败配置纠正 Agent、纠正提示和最大尝试次数。
2. 纠正 Agent 会收到原目标、上次结果、失败规则和允许使用的前序上下文。
3. 每次纠正后系统重新执行验收，成功后才进入后续节点。
4. 达到最大次数仍未通过时，流程进入用户配置的失败路线或人工接管状态。
5. 运行记录展示每次尝试的输入、结果、失败原因和纠正内容。
6. 有外部副作用的节点未声明幂等或未获得审批时，系统禁止自动重复执行。

## US-13 配置重试、备用能力和人工接管

**Card：** 处理自动化无法自行解决的问题

**Description：** 作为流程创建者，我希望为失败节点设置重试、备用能力或人工输入，以便流程在常见故障下仍能继续。

**Conversation：** 技术性临时错误适合重试，能力不可用适合备用节点，缺少业务判断时应暂停等待用户。恢复时不重复已完成节点。

**Design：** [运行恢复区域](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 用户可以配置最大重试次数、等待时间和适用错误类型。
2. 用户可以为 Agent、Skill 或 Tool 选择输入输出契约兼容的备用能力。
3. 节点需要信息或审批时，流程暂停并展示问题、上下文和可执行动作。
4. 用户提交输入、纠正或批准后，流程从暂停节点继续，不重复已完成节点。
5. 所有重试、切换、暂停和恢复操作都记录原因和执行者。

## US-14 通过定时或外部事件自动启动

**Card：** 无人值守触发流程

**Description：** 作为流程创建者，我希望按时间或外部事件自动启动流程，以便完整业务流程不依赖手工点击。

**Conversation：** 首批触发方式包括手动、定时和 Webhook。触发器只调用已发布流程版本，并需要明确处理重复和并发运行。

**Design：** 专用触发器设计待补，复用[流程运行入口](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 用户可以创建手动、定时和带密钥的 Webhook 触发器。
2. 定时触发器显示时区和下次运行时间，并可随时暂停或恢复。
3. Webhook 根据流程输入契约校验请求，不合法请求不会启动流程。
4. 用户可以选择允许并行、跳过重复或排队等待的并发策略。
5. 每次触发保存来源、时间、输入摘要、对应流程版本和运行结果。

## US-15 发布、复制和分享流程

**Card：** 管理可重复使用的流程版本

**Description：** 作为流程创建者，我希望把验证过的流程发布成固定版本，并能复制或导出分享，以便稳定复用和团队协作。

**Conversation：** 草稿可继续编辑，已发布版本不可被静默改变。导出包只包含流程和依赖声明，不包含凭据。

**Design：** 专用版本设计待补，复用[流程画布](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 用户可以保存草稿并发布带名称、版本和说明的流程版本。
2. 发布前系统执行完整预检，并固定使用的能力版本和输入输出契约。
3. 用户编辑已发布流程时，系统创建新草稿，不修改旧版本。
4. 用户可以复制流程作为新流程或保存为模板。
5. 用户可以导入导出流程包；包中不包含凭据，只包含所需能力和版本清单。
6. 用户可以将触发器回滚到以前的已发布流程版本。

## US-16 查看结果、重放和比较运行

**Card：** 运行历史与结果复用

**Description：** 作为流程创建者，我希望查找以前的运行并用相同输入重新执行，以便排查问题、复用结果和确认流程升级效果。

**Conversation：** 每次运行绑定流程版本、输入、能力版本和事件。重放默认使用原版本，比较用于识别新版本是否改变结果。

**Design：** [运行记录](../desktop/renderer/index.html)

**Acceptance Criteria：**

1. 用户可以按流程、状态、触发来源和时间筛选运行记录。
2. 运行详情展示流程版本、节点轨迹、纠正次数、人工操作和最终输出。
3. 用户可以下载结构化最终结果和脱敏后的运行证据。
4. 用户可以使用原流程版本和原输入重放一次运行。
5. 用户可以比较两个运行的节点状态、输出和耗时差异。
6. 数据达到保留期限前，系统提示用户导出或删除，不静默丢失记录。

## 与当前代码的关系

当前项目已有可复用的流程节点、状态分支、并行汇合、预检、重试、暂停恢复、运行事件和策略审批。需要新增或重做的核心产品能力是多来源导入、能力版本与契约、表单化数据映射、自动验收和纠正、触发器以及发布版本管理。
