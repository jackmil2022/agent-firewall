# Agent Firewall 2026 Q3 OKR 头脑风暴

## 项目判断

Agent Firewall 当前最可信的产品定位，不是提供强隔离保证的传统“防火墙”，而是面向 Agent、Skill 和 MCP Tool 作者的本地能力测试、诊断、修复与治理工作台。

项目已经具备较完整的 P0 产品骨架：能力发现、不可变测试快照、确定性断言、显式基线、候选修订、同快照比较、审核后应用、回滚、执行策略、审批和持久化前脱敏。当前自动化检查为 98 个 Python 测试和 3 个桌面端工作区测试，均通过；仓库中也已有可运行的 macOS arm64 打包产物。

当前最大缺口不是继续增加功能，而是证明这些能力在真实工作流中成立：项目本地数据库已有 24 次运行，但只有 1 个测试用例，且显式基线、比较、修订和 MCP Tool 发现记录均为 0。桌面端也尚未具备覆盖完整修复闭环的无头端到端发布门禁，测试用例、断言和修订仍主要依赖原始 JSON。

因此，Q3 最值得选择的战略方向有三个：建立可信发布闭环、证明治理有效性，或验证真实用户采用。季度内建议只选择一个 Objective，另外两组中的指标可作为健康指标或下一季度候选。

## 北极星指标与长期 KPI

候选北极星指标：**每周证据完整的能力验证闭环数**。一次闭环只有在目标与测试快照可核验、断言有明确结果，并最终形成有效基线或候选比较结论时才计数。

长期 KPI：首次价值时间、完整闭环成功率、策略绕过数、敏感信息落库数、周活跃能力作者数、每位作者验证的真实能力数。

## 方案 A：可信发布闭环

### Objective

在 2026 Q3 结束前，让 Agent Firewall 成为一个可以在干净环境中反复证明“测试、诊断、修复、审核、应用与回滚”结果可信的本地工作台。

### Key Results

1. 在 macOS 和 Windows 的干净打包环境中，对 Agent、Script Action、Skill Binding、MCP Tool、审批阻塞和取消恢复 6 类场景各重复 10 次，端到端闭环成功率达到 95% 以上，且证据快照不一致事件为 0。
2. 连续 10 个候选构建全部通过自动化发布门禁：创建测试、设置成功基线、运行修订候选、保存比较、完成审核、应用并回滚；阻断级缺陷为 0。
3. 邀请至少 5 名非项目开发者完成一次真实能力验证，首次建立可复用基线的中位时间不超过 15 分钟，至少 80% 的参与者无需开发者介入即可完成。

### 理由

这套方案优先兑现产品最核心的“证据可信”承诺，适合 Q3 目标是形成可交付 beta 或建立发布信心的情况。它不会把新增功能数量当作成功，而是衡量打包环境中的稳定性和用户完成结果。

## 方案 B：可验证的执行治理

### Objective

在 2026 Q3 结束前，让能力作者能够用可重复证据确认每一次 Agent、Skill 和 MCP 操作都遵守声明的资源边界、审批规则与敏感信息处理规则。

### Key Results

1. 对 Agent 直调、Flow、ACP、Script Action、MCP 连接/发现/调用、Agent 内嵌 MCP 调用和内置文件工具建立统一对抗样本集；200 次越权尝试的阻断率达到 100%，策略绕过为 0。
2. 在参数、结果、错误、事件、快照和导出数据中执行 100 个预置敏感信息泄漏场景，持久化或导出明文敏感值事件为 0。
3. 在至少 100 个安全与危险操作混合样本中，危险操作识别率达到 95% 以上，安全操作误拦截率低于 5%，需要审批的操作在批准后一次恢复成功率达到 99% 以上。

### 理由

这套方案强化“governed local execution”的可信度，适合需要面向平台、安全或合规团队证明治理价值的情况。它刻意不承诺容器或 VM 级强隔离；若产品对外声称安全沙箱，则需要另立隔离运行时目标。

## 方案 C：真实用户采用与产品学习

### Objective

在 2026 Q3 结束前，让一小批能力作者把 Agent Firewall 用于日常验证，并证明它比手工调试更快地形成可复用、可审核的能力证据。

### Key Results

1. 招募 8 名设计合作伙伴，其中至少 6 名在季度最后 4 周保持周活跃，每人累计验证至少 3 个真实 Agent、Skill Action 或 MCP Tool。
2. 至少 80% 的新用户在首次使用 20 分钟内完成“连接能力、创建测试、运行断言、设置基线”的首次价值路径，过程中无需编辑数据库或调用 CLI。
3. 累计完成至少 30 个证据完整的能力验证闭环，其中失败测试进入明确诊断并形成修订审核或放弃结论的比例达到 70% 以上。

### 理由

这套方案优先回答是否存在真实、重复的用户需求，适合项目尚未确定下一阶段产品边界时采用。它会自然暴露原始 JSON、能力接入、诊断可理解性和修订操作中的最大摩擦，再决定哪些功能值得继续投入。

## 取舍建议

- 选择方案 A：如果 Q3 的公司目标是发布 beta、提高交付可信度或建立跨平台发布门禁。
- 选择方案 B：如果公司目标是 Agent 风险治理、安全审查或受控执行能力建设。
- 选择方案 C：如果公司目标是产品发现、验证用户价值或避免继续在未验证需求上扩建。

无论选择哪套，建议保留三个健康指标：阻断级数据损失/泄漏事件为 0，主分支自动化检查通过率不低于 98%，产品文案始终明确“受治理的本地执行”不等于“强隔离沙箱”。

## 数据假设

- 当前本地 SQLite 数据只代表本仓库的开发活动，不能视为团队或外部用户使用基线。
- 项目目前没有正式埋点或跨工作区汇总机制；Q3 第 1-2 周需要先统一闭环、活跃用户、首次价值时间和策略事件的计算口径。
- Windows 打包闭环、真实 MCP 服务和真实模型供应商的稳定性尚未从现有测试与本地数据中得到证明。
